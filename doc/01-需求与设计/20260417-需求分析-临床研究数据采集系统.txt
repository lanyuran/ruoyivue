临床研究数据采集系统需求分析文档（结合 RuoYi-Vue 现状）

1. 文档信息
项目路径：`P:\Projects\ruoyivue`
文档版本：v1.0
编写日期：2026-04-16
分析目标：基于现有代码与数据库，形成可直接进入开发排期的需求分析文档（非通用模板）。

2. 项目现状基线（As-Is）

2.1 技术架构
后端：`Spring Boot 2.5.15 + MyBatis + Spring Security + Redis`
前端：`Vue2 + Element UI`
数据库：`MySQL`（README 约定库名 `fl`）
文件存储：`ruoyi.profile = ${user.dir}/image-cache`，文件上传通过 `/common/upload`，数据库保存相对路径/文件名。

2.2 已有业务模块
患者就诊数据模块：`/patient/information/*`
问卷模块：`/survey/survey/*`
用户体系扩展：医生注册申请与审批（`apply_role_key/apply_status`）
组织与权限扩展：`dept_doctor`、`dept_manager`、`campus_manager` 三层管理角色

2.3 已有关键数据表
`patient_visit_info`：就诊主表，包含 25 项问诊/检查字段及多张图片路径字段。
`sys_survey`：问卷定义（标题、内容 JSON、状态）。
`sys_user`：已扩展医生申请字段 `apply_role_key`、`apply_status`。

2.4 当前系统已实现能力
登录/注册（注册可选普通用户/医生申请）。
医生申请审批（通过/驳回）。
患者记录增删改查、Excel 导入、Excel 导出、图片上传。
按角色限制数据范围：医生/科室主管/院区主管。
问卷设计器（基于表单生成器）与问卷管理列表。
问卷列表中可生成二维码分享链接。

3. 核心问题与差距（Gap）

3.1 患者数据唯一性与数据质量
现状：`patient_visit_info.medical_record_no` 仅普通索引，未设置唯一约束；后端新增/导入未做重复拦截。
风险：同一病例号可重复写入，造成科研统计偏差。

3.2 问卷“分享-填写-回收”链路未闭环
现状：前端分享链接指向 `#/survey/fill/:id`，但项目中无 `survey/fill` 页面和提交答案后端接口。
风险：二维码可展示但无法完成真实外部填报。

3.3 角色层级与业务组织存在偏差
现状：角色为 `dept_doctor -> dept_manager -> campus_manager`，与“课题组跨院直接协作”场景存在流程冗余。
风险：审批与维护成本高，实际组织变化时权限配置复杂。

3.4 注册方式与临床使用习惯不一致
现状：登录名是 `username`，未强制手机号；没有短信找回密码能力。
风险：一线用户易忘账号，账号回收与支持成本高。

3.5 页面与终端体验
现状：患者相关页面为后台管理风格（大表格+弹窗），对手机端填报不友好。
现状：无“详情文档化打印”专用视图。
风险：家属填报门槛高，线下沟通与纸质归档效率低。

3.6 代码维护性
现状：`patient/information`、`patient/myInformation`、`check/information` 三套页面高度重复。
风险：需求变更需三处同步，容易产生行为不一致。

4. 需求目标（To-Be）

4.1 业务目标
支持“手机端快速填报 + 管理端审核/查询 + 可打印归档”的临床研究流程。
以病例号作为稳定主线，确保数据不重复、可追溯。
精简权限层级，支持跨院区课题管理。

4.2 产品目标
形成双端能力：
  - 采集端（H5）：面向家属/医生的轻量录入页。
  - 管理端（后台）：面向研究管理员的数据治理与审阅页。
问卷从“设计管理”升级为“可投放、可回收、可统计”。

5. 详细需求定义

5.1 账号与认证
5.1.1 手机号作为登录主键
注册时 `username` 由手机号承载（可与 `phonenumber` 同值）。
后台新增用户时默认同策略，避免账号体系双轨。
保留密码登录，不引入短信登录（本期）。

5.1.2 密码找回策略
本期：仅管理员重置密码（沿用现有 `/system/user/resetPwd`）。
下期：评估短信网关后再上“验证码找回”。

5.1.3 验收标准
新注册用户可直接使用手机号登录。
用户管理页支持按手机号快速检索。

5.2 角色与权限重构
5.2.1 角色模型调整
目标模型：`doctor`、`campus_manager`、`admin`（保留超管）。
`dept_manager` 角色下线（不再创建/分配，不影响历史数据读取）。

5.2.2 数据范围规则
`doctor`：默认仅本人创建数据（维持现策略）。
`campus_manager`：可跨院区查看与管理。
`admin`：全量权限。

5.2.3 配置与迁移
调整 SQL 初始化脚本中角色与 `sys_role_menu` 绑定逻辑。
服务层 `hasRole(ROLE_KEY_DEPT_MANAGER)` 分支改造并兼容历史角色。

5.2.4 验收标准
新系统不再出现 `dept_manager` 分配入口。
`campus_manager` 可管理多院区数据与用户审批。

5.3 患者采集端（新增 H5）
5.3.1 功能范围
新增独立 H5 录入页（移动优先），字段与 `patient_visit_info` 对齐。
支持图片上传（沿用 `/common/upload`）。
支持草稿与提交（本期至少支持提交）。

5.3.2 页面原则
单页分步录入（基础信息 -> 检查信息 -> 中医信息 -> 上传）。
大按钮、低认知负担、错误提示清晰。

5.3.3 权限方式
方案 A（推荐）：登录后访问。
方案 B（可选）：一次性口令链接，防止匿名滥填。

5.3.4 验收标准
主流手机尺寸（375/390/414 宽）可完整填报。
图片上传成功率与后台端一致。

5.4 病例去重与导入治理
5.4.1 去重规则
主规则：病例号不可重复。
冲突时返回明确错误（包括导入场景的失败明细）。

5.4.2 数据库与接口改造
增加唯一索引（建议 `uk_medical_record_no`）。
`insert`、`importData` 统一捕获重复键异常并转业务提示。

5.4.3 导入反馈
导入结果返回：成功条数、失败条数、失败原因（行号+病例号）。

5.4.4 验收标准
同病例号重复提交无法入库。
Excel 导入可定位到失败行并下载失败清单（可后续增强）。

5.5 详情页文档化与打印
5.5.1 详情展示
新增“文档阅读视图”（非表单弹窗样式），按问诊章节分组显示。
图片支持原图预览与打印占位。

5.5.2 打印能力
增加“打印”按钮，使用 `window.print()` + `@media print` 样式。
打印时隐藏操作区，仅保留患者信息、检查结果、图片索引、时间与责任人。

5.5.3 验收标准
A4 打印版式无明显错位。
记录可直接作为会诊/归档材料。

5.6 问卷模块闭环升级
5.6.1 保留现有能力
保留当前问卷设计器与问卷管理。

5.6.2 新增能力
新增 `survey/fill/:surveyId` 外部填写页（可匿名或轻认证）。
新增答案存储表（建议：`sys_survey_answer`、`sys_survey_answer_item`）。
新增提交接口、结果查询接口、导出接口。

5.6.3 分享能力优化
二维码生成优先使用本地生成方案，避免外部 `qrserver` 服务依赖。
分享链接带有效期与签名（防篡改/防刷）。

5.6.4 验收标准
从“问卷列表 -> 分享二维码 -> 手机填写 -> 后台查看结果”全链路可用。

5.7 管理端页面重构与复用
5.7.1 重构目标
合并三套重复页面为“单一业务组件 + 权限/模式差异配置”。

5.7.2 模式建议
模式1：我的数据（doctor）。
模式2：全局查询（manager/admin）。
模式3：只读审核（可选）。

5.7.3 验收标准
一个页面基座支持多角色差异显示，减少重复维护。

6. 非功能需求
安全：高危操作（删除/审批）保留审计日志。
可用性：移动端首屏 3 秒内可交互（内网目标）。
兼容性：Chrome/Edge/微信内置浏览器可用。
运维：2C4G 服务器可运行基础版本；图片目录可定期清理归档。

7. 数据与接口变更清单

7.1 数据库
`patient_visit_info`
  - 新增唯一索引：`medical_record_no`（或按业务改为 `hospital_dept_id + medical_record_no` 组合唯一）。
新增问卷答卷表（两张）。

7.2 后端接口
患者模块：新增去重校验与导入失败明细返回。
打印模块：可复用详情接口，前端做打印模板。
问卷模块：新增提交与结果查询接口。

7.3 前端页面
新增移动端采集页。
新增病例详情打印页。
新增问卷填写页与成功页。
合并重复的患者管理页面逻辑。

8. 里程碑建议（可执行）
M1（3-5天）：病例去重、角色简化、手机号登录名改造。
M2（4-6天）：详情文档化+打印、管理页复用重构。
M3（5-8天）：问卷填写闭环（填报+回收+导出）。
M4（3-5天）：移动端采集页、联调与灰度上线。

9. 风险与依赖
角色重构涉及历史用户与菜单数据迁移，需灰度方案。
病例号唯一规则需先确认“全局唯一”还是“院区内唯一”。
微信内访问若需企业能力（授权/菜单直达），依赖资质与配置。
外网部署需同步处理上传目录映射与访问权限。

10. 结论
当前 `ruoyivue` 已具备“后台录入 + 权限管理 + 问卷设计”的基础，但与临床实际使用目标相比，关键缺口在于：
移动端采集体验不足；
病例去重机制未闭合；
问卷仅能设计不能回收；
角色层级与组织流程尚未完全贴合。

按本分析文档推进后，可在现有架构上较低风险实现“可用、可扩展、可审计”的临床研究数据平台。
